Hooks. Automating the moments Claude can't see.

A hook is a small script that runs at a specific moment in the agent loop: before a tool runs, after a file is edited, when the session ends. The harness fires them, not Claude. That makes them dependable.

Why hooks exist

If you ask Claude "always run the linter after editing TypeScript files," sometimes Claude will, sometimes Claude won't. Memory drifts. Hooks make it a guarantee. The hook fires every time, regardless of what Claude decides.

The events

  • PreToolUse — fires before a tool runs. Can block.
  • PostToolUse — fires after. Good for "now lint this file."
  • UserPromptSubmit — fires when you press Enter on a message.
  • Stop — fires when the agent's turn ends.
  • SessionStart / SessionEnd — start and end of a session.

The shape of a hook

A hook is a shell script that reads JSON on stdin (info about what happened) and writes JSON on stdout (its decision: allow, block, modify). Configure them in .claude/settings.json:

{
  "hooks": {
    "PostToolUse": [
      {
        "matcher": "Edit",
        "command": "$CLAUDE_PROJECT_DIR/.claude/hooks/lint.sh"
      }
    ]
  }
}

Recipes worth stealing

  • Lint after edit. PostToolUse on Edit, runs your linter on the changed file.
  • Block dangerous shell commands. PreToolUse on Bash, refuses anything matching rm -rf.
  • Notify on stop. Stop hook, sends a desktop notification or Slack ping when a long task finishes.
  • Auto-format on edit. PostToolUse on Edit, runs Prettier on the file.

Best practices

Do

  • Reach for hooks when you want a guarantee, not guidance. Memory is guidance; hooks are enforcement.
  • Keep hook scripts fast. They run synchronously. Slow hooks slow every tool use.
  • Use $CLAUDE_PROJECT_DIR in paths so hooks work for any teammate who clones the repo.
  • Test hooks locally with sample JSON input before wiring them in.
  • Use .claude/settings.local.json while iterating. Move stable hooks to project settings to share.

Don't

  • Put long-running scripts in hooks. Network calls and big builds belong elsewhere.
  • Hide important information in hook output that you never read.
  • Use a hook where a CLAUDE.md rule would do. Reserve hooks for things you cannot trust to memory.
  • Forget hooks persist. A noisy hook adds noise to every session until you remove it.
  • Write hooks in languages you cannot debug. Bash is fine if you keep them small.
Newbie trap Misconfiguring a hook path, then wondering why nothing works. Use absolute paths or $CLAUDE_PROJECT_DIR. Run /doctor if a hook is silently failing.

Exercise. Add a Stop hook that prints "Session done at $(date)" to a log file. Wire it up in .claude/settings.local.json. Run a session, send any message, end the session. Check the log. You just instrumented Claude.
