Tokens, context, and the memory you forget you have.

Claude does not have memory the way you do. It has a window. Manage the window well and Claude feels brilliant. Manage it badly and Claude feels forgetful.

Tokens, in one paragraph

A token is roughly four characters of text. The word banana is two tokens. A page of writing is around 250. The "context window" is the total amount of text Claude can see in one turn: your message, the conversation so far, the files it has read, the tool outputs it has run. Sonnet and Opus give you 200,000 tokens by default, up to one million if you turn on extended context. Plenty for almost anything, until it isn't.

The two kinds of memory

Conversation memory. Lives only inside the current session. Closes when you start a new one (or when you run /clear).

Persistent memory. Survives sessions. Two flavors:

  • CLAUDE.md — a file you write, sitting at the top of your project. Loaded into every session in that folder. Use it for facts that never change: build commands, conventions, who to ask, where things live.
  • Auto-memory — a file Claude writes for you in ~/.claude/projects/<your-project>/memory/. Claude saves things it learns about you over time. You can read it (run /memory), edit it, or delete entries.

The four commands you will actually use

/init — once per project, at the start. Claude reads the codebase and drafts a starter CLAUDE.md for you. Edit it. Commit it.

/memory — open the memory files (your CLAUDE.md and the auto-memory). Edit in place. Add a fact, remove a stale one.

/context — show what is currently filling your context window: rules, files, tools, conversation. Run this if responses start feeling slow or vague.

/compact focus on X — when context fills up, this summarizes the conversation so far while keeping the spine you specified. "/compact focus on the auth refactor" drops the digressions, keeps the work.

Auto-compaction (the safety net)

You will not hit "out of context" errors. When the window gets full, Claude Code automatically summarizes old conversation, drops verbose tool outputs, and re-injects your CLAUDE.md and the active task. You will see a small banner saying so. The lesson: put your durable rules in CLAUDE.md, not in chat. Chat gets summarized away.

Best practices

Do

  • Run /init the moment you open a new project. Future sessions get smarter for free.
  • Keep CLAUDE.md short and factual. Aim under 200 lines.
  • Use nested CLAUDE.md files inside subfolders for scope-specific rules. They load only when relevant.
  • Check /context when responses get sluggish or off-topic.
  • Use /compact focus on … before context hits 80%, naming what to preserve.
  • Use /clear when switching to a totally different task in the same project.
  • Read /memory every week or two. Prune outdated facts.

Don't

  • Treat CLAUDE.md as a notebook. It is a contract. Vague entries get ignored.
  • Paste long reference material into chat hoping Claude "remembers" it next session. Put it in a file.
  • Run /compact at 30% context. You will lose detail you still needed.
  • Stuff every preference into one massive CLAUDE.md. Split rules into files or skills.
  • Forget that auto-memory persists across sessions. If Claude learned something wrong, fix it.
  • Repeat instructions in every prompt that already live in CLAUDE.md. Trust the file.
Newbie trap Editing CLAUDE.md mid-session and expecting the change to apply now. It applies to the next session. End the session, start a new one, and the new rule is live.

Exercise. In your elad-test folder, run /init and let Claude draft a CLAUDE.md. Open the file. Add one line: "My name is Elad. Always greet me by name in the first response of a session." Save. Close the session. Open a new one. Send "hi." Did it greet you?
