Git for agents. Your safety net and your time machine.

You do not need to be a git expert to use Claude Code well. You need to know two things: commit before you experiment, and use worktrees when you want parallel agents.

Why git matters here

Agents move fast. They edit many files. Sometimes they go in a direction you do not want. Git is the undo button. With one clean commit before you start, every wrong turn is a git reset away from gone.

If you do not yet use git, this is the time. git init in any folder, git add . and git commit -m "first" gives you a clean snapshot. Now Claude can experiment freely.

Letting Claude commit

Claude can commit on your behalf. Just ask: "commit these changes with a clear message." Add specific git commands to your permissions.allow so it does not prompt every time:

"Bash(git status)",
"Bash(git diff *)",
"Bash(git add *)",
"Bash(git commit *)"

Leave git push in ask or omit it. Pushing is a public action; you should look first.

Worktrees: parallel agents in one repo

A worktree is a separate folder on your filesystem checked out to a different branch of the same repository. Two folders, two branches, two isolated workspaces. Two Claude Code sessions can run in parallel without colliding.

git worktree add ../elad-test-experiment experiment-branch

Now you have elad-test/ on main and elad-test-experiment/ on the experiment branch. Open Claude Code in each. They cannot collide.

What to gitignore

Add to .gitignore:

.env
.env.local
node_modules/
.DS_Store
.claude/settings.local.json

Anything with secrets, anything machine-specific, anything generated. If Claude can see it, Claude might suggest changes to it. Hide what should be hidden.

Best practices

Do

  • Commit before you start an agentic task. Always.
  • Whitelist specific git commands in settings. Reduce prompts without granting blanket access.
  • Use worktrees when you want to run two Claude sessions on the same repo.
  • Let Claude write commit messages. They are usually clear and consistent.
  • Run git status before any big agent task. Clean state = clean diff at the end.

Don't

  • Allow Bash(git *) as a blanket. git push --force can erase team history.
  • Run two Claude sessions in the same folder on the same branch. Use a worktree.
  • Leave secrets out of .gitignore. .env files end up in commits this way.
  • Accept commits without reading the diff. /diff shows everything; review it.
  • Force-push to shared branches without warning the team. Even if Claude does it for you.
Newbie trap Confusing git branches with git worktrees. A branch is a label on a commit. A worktree is a physical folder containing files. Two branches, one folder = you switch and lose state. Two worktrees, two folders = you keep both alive.

Exercise. In elad-test, run git init and commit your existing files. Now ask Claude: "create a git worktree at ../elad-test-experiment on a new branch called experiment, then open it for me." You now have two folders, two branches, ready for two parallel agents.
