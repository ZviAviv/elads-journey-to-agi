Subagents. Delegating work without bloating context.

A subagent is a fresh Claude with its own context window, sent off to do one task. It comes back with a summary. Your main conversation stays clean.

Why subagents exist

Some tasks generate a lot of intermediate noise: digging through hundreds of files, reading huge logs, running long research. If that noise lives in your main session, it crowds out the work you actually care about. A subagent contains it.

How they work

You define a subagent (or use one of the bundled ones). When you ask Claude to do a task that matches the subagent's description, Claude spawns a copy with that subagent's instructions, gives it the task, and waits. The subagent runs in its own context window. It returns a result. Your main session sees only the result, not the journey.

Defining one

Run /agents to manage subagents in the Desktop app. Each has a name, a description, an allowed tool list, and a system prompt. Bundled ones include things like Explore (read-heavy codebase exploration) and Plan (write a plan, do not edit).

When to delegate

Use a subagent when:

  • The task generates output you will not re-read (search results, log digests).
  • The task is recurring and worth standardizing ("review every PR for security").
  • The task should not pollute the main conversation (research that produces a one-line answer).

Skip a subagent when the task is fast and the result matters in context. Spawn overhead is real.

Best practices

Do

  • Use Explore subagents for "find all the places X happens" — they search and summarize.
  • Brief subagents fully. They do not see your conversation. Give them everything they need in the task prompt.
  • Restrict each subagent's tools to what it actually needs. A research subagent needs Read and Grep, not Edit.
  • Define subagents for recurring patterns: "run the security review subagent before any merge."
  • Let Claude pick the subagent automatically when the description matches your request.

Don't

  • Spawn subagents for trivial tasks. Spawn overhead is greater than the work.
  • Give subagents broad tool access. Wider access = wider blast radius.
  • Define ten subagents you might use. Define the two you actually will.
  • Forget that subagents do not share memory with each other or with the main session.
  • Confuse subagents with parallel sessions. Subagents return; sessions persist.
Newbie trap Briefing a subagent like you brief Claude in a chat: vague, hand-wavy, assuming context. Subagents start cold every time. The prompt has to stand on its own.

Exercise. In any project with more than ten files, ask: "use the Explore subagent to find every place the word 'TODO' appears in this codebase, then summarize what kind of work is left." Watch the main conversation stay short while the subagent does the digging.
