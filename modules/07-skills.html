Skills. Teaching Claude your way of working, once.

A skill is a small Markdown file that tells Claude how to do a recurring task. The first time you paste the same instructions twice, you should be writing a skill instead.

What a skill looks like

A skill is one folder with one file in it: SKILL.md. The folder name becomes the slash command. So ~/.claude/skills/code-review/SKILL.md creates /code-review.

Inside SKILL.md, you write YAML frontmatter and then Markdown instructions:

---
name: code-review
description: Review staged changes, flag bugs and risky patterns, suggest fixes.
---

When this skill runs, do the following:

1. Run `git diff --staged` to see what changed.
2. For each file, check for: missing error handling, magic numbers,
   dead code, security risks.
3. Write a report grouped by file with line references.

Where they live

  • Personal: ~/.claude/skills/ — available in every project on your machine.
  • Project: .claude/skills/ inside a project — shared with anyone who clones the repo.
  • Plugin: distributed inside a plugin (more in module 09).

How they get invoked

Two ways. Either you type /code-review directly, or Claude reads the description in your frontmatter and triggers the skill itself when your request matches. That description is doing real work — it is the trigger. Make it specific.

If you want to control timing (deploys, commits, anything with side effects), set disable-model-invocation: true in the frontmatter. Now Claude will not auto-invoke. Only you can.

Skill vs slash command vs CLAUDE.md vs subagent

CLAUDE.md: facts and conventions. "Always use pnpm."

Skill: a procedure. "Here is how I review code."

Slash command: in modern Claude Code, slash commands and skills are the same thing — typing /foo runs the foo skill.

Subagent: a separate context to delegate work to (module 13).

Best practices

Do

  • Write a skill the second time you paste the same instructions.
  • Make the description specific and verb-first. "Deploy to staging" beats "deploy stuff."
  • Set disable-model-invocation: true for anything with side effects.
  • Keep SKILL.md short. Move long reference material into separate files in the skill folder, link from SKILL.md.
  • Commit project-level skills to git. They are team config.
  • Iterate live. Edit a skill mid-session and the changes apply on next invocation.

Don't

  • Stuff one mega-skill with eight unrelated tasks. Split it.
  • Write vague descriptions. Vague description = skill never auto-triggers.
  • Put everything in CLAUDE.md when it should be a skill. CLAUDE.md is for what is true; skills are for what to do.
  • Forget to test a new skill. Type its slash command and watch what happens.
  • Build skills for things Claude already does well unprompted. You are adding overhead, not capability.
Newbie trap The full body of a skill does not load into context until the skill is invoked. Only the description loads at session start. That is the whole point — skills stay cheap until you need them. Write skills freely.

Exercise. Make a skill called journal-today in .claude/skills/journal-today/SKILL.md inside your elad-test project. Description: "Create a markdown journal entry in journal/ for today's date with a heading and three bullet points for things learned." Body: a short procedure. Save. In Claude Code, type /journal-today. Did it run?
