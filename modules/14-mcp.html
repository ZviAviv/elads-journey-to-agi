MCP. The way Claude reaches into your other tools.

MCP stands for Model Context Protocol. It is the open standard that lets Claude talk to external services: GitHub, Slack, Linear, Google Drive, your database. Add an MCP server, and Claude grows new abilities.

The core idea

Claude Code ships with tools for files, the shell, and the web. That covers most of what you need locally. But the world has many other systems with their own APIs. MCP is how Claude talks to them in a consistent way.

An MCP server runs alongside Claude Code (locally or remotely). It exposes tools that Claude can call: "send a Slack message," "create a Linear ticket," "query a Postgres database." From Claude's point of view, they are just more tools.

How to add one

Browse the official registry at registry.anthropic.com or run /mcp to add and manage servers. Each server has a name, a connection method, and usually some configuration (API keys, URLs).

Servers are namespaced. A GitHub MCP server might expose a tool called mcp__github__create_issue. The prefix prevents clashes.

What MCP is not

MCP is not for adding local powers — file editing, shell commands, web fetches all already exist. MCP is for reaching out to systems Claude does not natively know about.

MCP is also not the same as plugins. Plugins are bundles of Claude Code config (skills, hooks, agents). Some plugins include MCP servers; others do not. Plugins are how things get distributed; MCP is one of the things they can distribute.

Best practices

Do

  • Add an MCP only when you have a specific service you want Claude to act on regularly.
  • Read the auth and permission requirements before installing. Some need broad scopes.
  • Use tool search to defer loading MCP tool definitions until they are needed. Saves startup context.
  • Prefer official MCPs from the registry. Community ones can be great but vary in quality.
  • Rotate API keys regularly. MCP credentials are still credentials.

Don't

  • Install ten MCPs in case you might need them. Each one adds startup time and context cost.
  • Give MCP servers more permission than they need. Read-only is usually fine for early experimentation.
  • Assume MCPs are free. Many sit on top of paid APIs. Watch your usage.
  • Confuse MCP with subagents. Different mechanisms. Different problems.
  • Trust unofficial MCPs blindly. Inspect what they do before granting credentials.
Newbie trap Installing the GitHub MCP and the GitHub plugin and the gh CLI all at once, then losing track of which one Claude is using. Pick one path. Fewer tools doing the same job is calmer.

Exercise. Open /mcp. Browse what is in the registry. Pick one service you actually use (Slack, GitHub, Notion). Read what it can do and what it asks for. Do not install yet. The decision is the exercise.
