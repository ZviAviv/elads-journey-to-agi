Permissions. Less prompt fatigue, no less safety.

Every confirmation dialog is a tax on your attention. Settings let you pre-approve trusted moves and pre-block scary ones, so the prompts you do see are the ones that matter.

Where settings live

  • ~/.claude/settings.json — your personal defaults. Apply everywhere.
  • .claude/settings.json — project-level. Shared with anyone who clones.
  • .claude/settings.local.json — your local overrides for one project. Gitignored by default.

Precedence: managed > local > project > user. Most specific wins.

The shape of a permission rule

{
  "permissions": {
    "allow": [
      "Bash(npm test)",
      "Bash(npm run build)",
      "Bash(git status)",
      "Bash(git diff *)"
    ],
    "deny": [
      "Edit(~/.ssh/**)",
      "Bash(rm -rf *)"
    ],
    "ask": [
      "Bash(git push *)"
    ]
  }
}

Three buckets: allow (run without asking), deny (refuse always), ask (prompt every time). Deny beats allow.

Strategy

Start strict. After a week, look at the prompts that bored you. Move those into allow. Look at the prompts that scared you. Make them deny. Iterate.

Specific rules beat broad ones. Bash(npm run build) is safer than Bash(npm *). The narrower the allow, the less surprise.

Default mode in settings

You can lock in your starting mode (Module 05) with "defaultMode": "acceptEdits" in user settings. Saves the Shift+Tab dance every session.

Best practices

Do

  • Run /permissions to see what is allowed before tweaking.
  • Add specific commands to allow as you notice repeat prompts. Whitelist the boring stuff.
  • Always deny destructive commands explicitly. rm -rf, force pushes, anything irreversible.
  • Keep project-level settings checked into git so the team shares them.
  • Use .claude/settings.local.json for your personal overrides; it is gitignored.

Don't

  • Allow broad patterns to silence prompts. Bash(*) in allow is how disasters happen.
  • Set bypassPermissions on your work machine. Sandboxes only.
  • Forget the precedence order. Local wins over project. Project wins over user.
  • Grant network access to plugins or MCPs that should not need it.
  • Edit settings while a session is running and expect the change to apply mid-session. Restart.
Newbie trap Allowing yourself out of every prompt and then walking away while Claude works. Permissions reduce friction; they do not replace your attention. Stay in the loop.

Exercise. Open ~/.claude/settings.json (create it if missing). Add an allow rule for Bash(ls *) and a deny rule for Bash(rm -rf *). Save. Restart Claude Code. Ask it to list files in your project. No prompt. Now ask it to delete everything. It refuses. Both behaviors are now baked in.
