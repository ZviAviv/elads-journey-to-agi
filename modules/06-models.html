Part 2 · The Engine 06

Models. Three Claudes. Pick the right one.

Opus thinks hardest. Sonnet works hardest. Haiku moves fastest. The trick is not picking once. It is switching mid-session.

The lineup (2026)

  • Opus 4.7. The deepest reasoner. Best for architecture, novel problems, tricky bugs, anything where being right matters more than being fast. Most expensive per token.
  • Sonnet 4.6. The default. Fast enough for daily work, smart enough for almost everything you will do in a day. Best balance.
  • Haiku 4.5. Lightning fast, very cheap. Great for simple edits, generating boilerplate, running tests, summarizing logs.

The aliases worth knowing

/model sonnet picks the latest Sonnet. /model opus picks the latest Opus. /model haiku picks the latest Haiku. /model opusplan uses Opus while planning and switches to Sonnet to execute. Best of both worlds for big tasks.

Fast mode (toggle with /fast) runs Opus 4.6 instead of 4.7 — slightly less capable, noticeably quicker, only available while on Opus.

How to think about cost

You are not paying for the conversation, you are paying for tokens. Opus on a long-context refactor can be ten times the cost of the same refactor on Sonnet. Sonnet is usually the right answer. Reach for Opus when you have a hard problem and reach for Haiku when you have a small one.

Effort, the second dial

/effort tunes how hard the model thinks per turn. Default is xhigh on Opus. Drop to high or medium for cost savings on routine work. Crank to max for one truly hard problem, then turn it back down.

Best practices

Do

  • Default to Sonnet. Switch only when the task tells you to.
  • Use opusplan for big tasks. Opus to plan, Sonnet to do.
  • Drop to Haiku for repetitive work — formatting, tests, small fixes.
  • Switch mid-session with /model. No restart needed.
  • If you keep hitting context limits in a huge codebase, try opus[1m] or sonnet[1m] for the million-token window.

Don't

  • Use Opus for everything. You will burn through budget and gain little.
  • Use Haiku for novel architectural decisions. Wrong tool.
  • Crank effort to max and forget it. Reset after the hard turn.
  • Assume the model picks itself based on the prompt. You pick.
  • Switch models five times in one task. Decide and commit.
Newbie trap Confusing "model" and "effort." Model is which Claude. Effort is how hard that Claude thinks. Both matter. Opus on low effort can be weaker than Sonnet on high.

Exercise. Pick a moderately hard question — for example, "design a folder structure for a personal newsletter project where I write monthly issues, each with images, drafts, and a published version." Ask Sonnet. Read the answer. Run /model opus. Ask the same question. Compare.
