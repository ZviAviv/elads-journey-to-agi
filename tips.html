Tips & best practices.

Every Do and Don't from across the guide, in one place.

When you forget which mode to use, when to /compact, or whether to write a skill or a hook — here it is. Grouped by topic. Searchable.

Module 01 · What's an agent?

Agents vs chat

Do

  • Reach for an agent when the task touches files, commands, or multiple steps.
  • Give context up front: file paths, error messages, what success looks like.
  • Expect the agent to ask questions back. Answer them.
  • Let the agent iterate. Watch the loop. Course-correct early.
  • Use chat for things with no side effects: explain, brainstorm, summarize.

Don't

  • Treat the agent like a search engine.
  • Send vague prompts and hope.
  • Walk away while the agent works in your real folders.
  • Assume the agent already knows your project on the first prompt.
  • Paste a 200-line file into chat to get help editing it.
Module 02 · Code vs Cowork vs Chat

Picking the right surface

Do

  • Default to chat for "explain this" or "what would you do."
  • Use Cowork when work lives in Drive, Gmail, or a spreadsheet.
  • Use Claude Code when work lives in a folder on your machine.
  • Move work between surfaces deliberately.
  • Bookmark all three. Friction kills the habit.

Don't

  • Paste your codebase into chat.
  • Try to build a real product in Cowork.
  • Open Claude Code to ask "what is a closure."
  • Assume one surface knows what you did in another.
Module 03 · Your first prompt

Writing prompts that work

Do

  • Lead with the goal, then constraints, then details.
  • Paste error messages and stack traces verbatim.
  • Ask for a plan before any change you would not want to undo.
  • Reference patterns: "use the same style as Button.tsx."
  • Tell the agent what NOT to do, not just what to do.
  • End with success criteria. One sentence is enough.
  • Show examples or screenshots when the work is visual.

Don't

  • Write a prompt like a Slack message. Specificity is kindness.
  • Stack ten asks into one prompt.
  • Assume the agent remembers yesterday's session.
  • Bury the key requirement in the third paragraph.
  • Apologize or pad. It does not improve results.
  • Send "fix it" without saying what "it" or "fixed" mean.
Module 04 · Tokens & memory

Managing context and memory

Do

  • Run /init the moment you open a new project.
  • Keep CLAUDE.md short and factual. Aim under 200 lines.
  • Use nested CLAUDE.md files for scope-specific rules.
  • Check /context when responses get sluggish.
  • Use /compact focus on … before context hits 80%.
  • Use /clear when switching to a totally different task.
  • Read /memory every week or two and prune.

Don't

  • Treat CLAUDE.md as a notebook. It is a contract.
  • Paste long reference material into chat hoping Claude remembers.
  • Run /compact at 30% context.
  • Stuff every preference into one massive CLAUDE.md.
  • Forget that auto-memory persists across sessions.
  • Repeat instructions in every prompt that already live in CLAUDE.md.
Module 05 · Modes

Choosing the right mode

Do

  • Use Plan mode before any change you cannot easily undo.
  • Cycle modes mid-session with Shift+Tab.
  • Drop to Ask permissions in production code.
  • Plan mode + Opus is one of the strongest combos.
  • Set defaultMode: "acceptEdits" in settings if you trust yourself.

Don't

  • Use Bypass on your laptop. Ever.
  • Stay in Plan mode and wonder why nothing happens.
  • Leave Auto Accept on overnight.
  • Switch to Auto in a repo you have not committed cleanly.
  • Treat the modes as a setting you set once.
Module 06 · Models

Picking models

Do

  • Default to Sonnet. Switch only when the task tells you to.
  • Use opusplan for big tasks.
  • Drop to Haiku for repetitive work.
  • Switch mid-session with /model.
  • Try opus[1m] or sonnet[1m] for huge codebases.

Don't

  • Use Opus for everything.
  • Use Haiku for novel architectural decisions.
  • Crank effort to max and forget it.
  • Assume the model picks itself based on the prompt.
  • Switch models five times in one task.
Module 07 · Skills

Writing skills

Do

  • Write a skill the second time you paste the same instructions.
  • Make the description specific and verb-first.
  • Set disable-model-invocation: true for side effects.
  • Keep SKILL.md short; move long material to separate files.
  • Commit project-level skills to git.
  • Iterate live. Edits apply on next invocation.

Don't

  • Stuff one mega-skill with eight unrelated tasks.
  • Write vague descriptions.
  • Put procedures in CLAUDE.md when they should be skills.
  • Forget to test a new skill.
  • Build skills for things Claude already does well unprompted.
Module 08 · Slash commands

Daily slash commands

Do

  • Run /help the first time you open the app.
  • Use /rename on any session you might want to find again.
  • Use /context proactively when responses get sluggish.
  • Mental shortcut: /clear for new task, /compact to continue.
  • Use /doctor first when something feels broken.

Don't

  • Type /clear thinking you are deleting history.
  • Memorize commands you never use.
  • Use /compact at low context.
  • Run /init on a project with a customized CLAUDE.md.
  • Forget arguments after the slash are a thing.
Module 09 · Plugins

Installing plugins

Do

  • Stick to the official marketplace until you have a reason not to.
  • Read what tools the plugin requests.
  • Install one plugin at a time.
  • Uninstall plugins you stop using.
  • Look at well-made plugins as study material.

Don't

  • Install plugins for things Claude Code already does.
  • Pile up plugins to feel productive.
  • Trust unsigned plugins from random repos.
  • Forget plugins persist and load at startup.
  • Confuse plugins with MCP servers.
Module 10 · Sessions

Managing sessions

Do

  • Rename any session you might want to find again.
  • /clear when switching to an unrelated task in the same folder.
  • Use git worktrees for true parallel work in one repo.
  • Use /branch for "what if" experiments without losing thread.
  • Treat /resume as your save game menu.

Don't

  • Run two Desktop sessions on the exact same folder.
  • Let one session run forever.
  • Delete ~/.claude/projects/.
  • Forget you have a session running in another window.
  • Use /branch when you actually need a worktree.
Module 11 · Permissions

Settings & permissions

Do

  • Run /permissions to see what is allowed.
  • Add specific commands to allow as you notice repeat prompts.
  • Always deny destructive commands explicitly.
  • Keep project-level settings checked into git.
  • Use .claude/settings.local.json for personal overrides.

Don't

  • Allow broad patterns to silence prompts.
  • Set bypassPermissions on your work machine.
  • Forget the precedence: local > project > user.
  • Grant network access to plugins that should not need it.
  • Edit settings mid-session and expect changes to apply.
Module 12 · Git basics

Git for agents

Do

  • Commit before you start an agentic task. Always.
  • Whitelist specific git commands in settings.
  • Use worktrees for two Claude sessions on the same repo.
  • Let Claude write commit messages.
  • Run git status before any big agent task.

Don't

  • Allow Bash(git *) as a blanket.
  • Run two Claude sessions in the same folder on the same branch.
  • Leave secrets out of .gitignore.
  • Accept commits without reading the diff.
  • Force-push to shared branches without warning the team.
Module 13 · Subagents

Delegating to subagents

Do

  • Use Explore subagents for "find all the places X happens."
  • Brief subagents fully. They do not see your conversation.
  • Restrict each subagent's tools to what it needs.
  • Define subagents for recurring patterns.
  • Let Claude pick the subagent automatically when descriptions match.

Don't

  • Spawn subagents for trivial tasks.
  • Give subagents broad tool access.
  • Define ten subagents you might use. Define the two you actually will.
  • Forget subagents do not share memory.
  • Confuse subagents with parallel sessions.
Module 14 · MCP

Adding MCP servers

Do

  • Add an MCP only when you have a specific service to act on regularly.
  • Read the auth and permission requirements first.
  • Use tool search to defer loading MCP definitions.
  • Prefer official MCPs from the registry.
  • Rotate API keys regularly.

Don't

  • Install ten MCPs in case you might need them.
  • Give MCP servers more permission than they need.
  • Assume MCPs are free.
  • Confuse MCP with subagents or plugins.
  • Trust unofficial MCPs blindly.
Module 15 · Hooks

Writing hooks

Do

  • Reach for hooks when you want a guarantee, not guidance.
  • Keep hook scripts fast.
  • Use $CLAUDE_PROJECT_DIR for portable paths.
  • Test hooks locally with sample JSON input.
  • Use .claude/settings.local.json while iterating.

Don't

  • Put long-running scripts in hooks.
  • Hide important information in hook output you never read.
  • Use a hook where a CLAUDE.md rule would do.
  • Forget hooks persist.
  • Write hooks in languages you cannot debug.
Module 16 · Claude Design vs Code

Picking Design or Code

Do

  • Use Claude Design for anything that lives as pixels.
  • Use Claude Code for anything that lives as files.
  • Hand off intentionally: design first, then build.
  • Open both apps if your work alternates.

Don't

  • Try to build production software in Claude Design.
  • Try to design a brand in Claude Code.
  • Confuse Claude Design with chat Artifacts.
  • Assume one will replace the other.
